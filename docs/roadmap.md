## 📍 当前位置

Hot Hub 已完成 **数据采集层 + 审查检测层 + 规则评分引擎**，在"热搜消失检测"这个垂直方向上做得比较扎实。但对比 BettaFish 等舆情系统，我们在**深度分析、智能化、评论维度、报告输出**上还有明显差距。

## 🗺️ Roadmap

### Phase 1：夯实基础 + 轻量智能（1-2 周）

> 目标：不依赖 GPU/大模型，用规则+统计+轻量NLP 把现有体系补完

#### 1.1 降温曲线检测（纯统计）
- **思路**：每次快照记录每个条目的 `(timestamp, rank)`，构建排名时序
- **正常降温**：排名单调递增（越来越靠后），斜率平缓
- **审查删除**：排名稳定/上升中突然消失（最后一个数据点 → 无）
- **实现**：线性回归斜率 + 最后 N 点方差，无需 ML
- **产出**：`DissipationCurveDetector` 从占位变为可用

#### 1.2 轻量情感分析（SnowNLP）
- **方案**：SnowNLP，纯 Python，无 GPU，中文情感打分 0-1
- **集成**：`SentimentAnalyzer.analyze()` 接入 SnowNLP
- **局限**：精度一般，但零成本启动，后续可替换为 BERT
- **依赖**：`pip install snownlp`

#### 1.3 轻量实体识别（jieba + 规则）
- **方案**：jieba 分词 + 自定义敏感实体词典（官员/机构/地名）
- **集成**：`EntityExtractor.extract()` 用 jieba 提取人名/地名
- **词典**：维护 `data/sensitive_entities.json`
- **依赖**：`pip install jieba`

#### 1.4 热榜趋势对比（前端图表）
- **思路**：从 `hot_item` 历史数据中提取某条目的排名变化曲线
- **前端**：用 Chart.js 或 ECharts 画折线图
- **入口**：点击任意热搜条目 → 弹出趋势图
- **API**：`/api/trend?title=xxx&platform=xxx`

#### 1.5 关键词搜索/过滤
- **前端**：搜索框，支持在审查监测和主页搜标题
- **后端**：SQL LIKE 或 FTS5 全文索引

### Phase 2：评论维度 + 深度分析（2-4 周）

> 目标：从"只看标题"升级到"看评论"，这是舆情分析的核心跳跃

#### 2.1 评论采集引擎
- **难点评估**：
  - 微博评论：需要 Cookie/登录态，反爬严格，Playwright 模拟可行但慢
  - 知乎回答：API 相对友好，`/api/v4/questions/{id}/answers`
  - 百度贴吧：评论结构简单但需适配
  - 抖音/小红书：重度 JS 渲染，必须 Playwright
- **策略**：**只对"高度可疑"条目（score >= 70）自动触发**，控制爬取量
- **架构**：
  ```
  censor 检测到高可疑 → 丢入评论采集队列 → 异步 Playwright 爬取
  → 存入 comment_item 表 → 触发情感分析
  ```
- **存储**：新增 `comment_item` 表（platform, topic_title, content, author, time, sentiment）
- **优先级**：微博 > 知乎 > 百度 > 其他

#### 2.2 评论情感聚合
- **流程**：采集到评论 → 逐条跑情感分析 → 聚合为话题级民意方向
- **输出**：
  ```json
  {
    "topic": "某市官员落马",
    "comment_count": 156,
    "sentiment": { "positive": 0.12, "negative": 0.73, "neutral": 0.15 },
    "top_negative_comments": [...],
    "top_positive_comments": [...]
  }
  ```
- **前端**：在审查监测卡片下方展示民意饼图 + 代表性评论

#### 2.3 语义话题聚类
- **方案**：`sentence-transformers` (paraphrase-multilingual-MiniLM-L12-v2) + HDBSCAN
- **流程**：embed 所有消失标题 → 聚类 → 同簇视为同一话题
- **替代当前**：字符串匹配 → embedding 语义匹配
- **依赖**：`pip install sentence-transformers hdbscan`（需要约 500MB，CPU 可跑）

#### 2.4 LLM 辅助摘要（可选）
- **触发**：高可疑条目 → 调用 LLM 生成一句话总结"为什么可疑"
- **方案**：调本地 API 代理（已有 `http://127.0.0.1:8080/api/openai/v1`）
- **成本控制**：只对 score >= 70 的条目调用，每天预估 < 50 次

### Phase 3：报告 + 告警 + 可视化（4-6 周）

> 目标：从"看数据"升级到"出报告"

#### 3.1 每日舆情简报
- **自动生成**：每天 08:00 生成前 24h 审查监测简报
- **内容**：Top 10 可疑事件 + 跨平台联动事件 + 民意方向 + 趋势图
- **格式**：HTML 邮件 / Markdown / 飞书文档
- **推送**：飞书机器人 / Discord webhook / 邮件

#### 3.2 实时告警
- **触发条件**：
  - 可疑度 >= 80 的条目出现
  - 3+ 平台联动消失事件
  - 高热条目（Top 3）在 10 分钟内消失
- **推送**：飞书 / Discord / Webhook
- **降噪**：同一话题 1 小时内只告警一次

#### 3.3 数据可视化大屏
- **方案**：独立页面 `/dashboard`
- **内容**：
  - 24h 消失数量趋势线
  - 平台消失分布饼图
  - 可疑度分布直方图
  - 跨平台联动网络图
  - 实时滚动 feed
- **技术**：ECharts / D3.js

#### 3.4 暗色模式
- CSS `prefers-color-scheme` + 手动切换按钮

### Phase 4：模型升级 + 高级分析（6-12 周）

> 目标：接入真正的 NLP 模型，从"规则"升级到"智能"

#### 4.1 BERT 敏感度分类器
- **方案**：微调 `bert-base-chinese` 在敏感内容数据集上做二分类
- **数据**：从历史 censored_item 中标注"确认审查" vs "正常降温"
- **部署**：ONNX 量化，CPU 推理 < 50ms/条
- **替代**：当前的关键词规则引擎

#### 4.2 升级情感分析
- **方案**：从 SnowNLP 升级到 `hfl/chinese-roberta-wwm-ext` 微调版
- **或者**：调用 LLM 做 few-shot 情感分类（精度最高，成本较高）

#### 4.3 降温曲线 ML 版
- **数据**：积累足够的排名时序后（2-3 个月数据）
- **方案**：LSTM / Transformer 小模型，输入排名序列，输出 {审查, 自然} 概率
- **训练**：历史数据 + 人工标注

#### 4.4 多 Agent 深度分析（参考 BettaFish）
- **评估**：对于 Hot Hub 的定位，完整的多 Agent 系统可能过重
- **轻量版**：单个 LLM Agent，给定话题 + 评论数据 → 输出分析结论
- **如果需要**：可参考 BettaFish 的 ForumEngine 做 Agent 间辩论

## 📊 复盘检查清单（每日 02:00）

每次复盘回答以下问题：

1. **数据质量**：过去 24h 有多少平台爬取失败？哪些需要修复？
2. **检测效果**：可疑度评分是否合理？有没有明显的 false positive/negative？
3. **噪音过滤**：有没有漏过的广告/营销内容？需要补充关键词？
4. **性能**：爬取耗时、数据库大小、内存占用是否正常？
5. **Roadmap 进度**：当前在哪个 Phase？下一步该做什么？
6. **业界动态**：有没有新的开源项目/论文/技术值得借鉴？

## 🎯 项目定位

**Hot Hub ≠ BettaFish**

- BettaFish 是"通用舆情分析平台"，重交互式 Agent + 报告生成
- Hot Hub 是"审查监测工具"，重**实时性 + 检测精度 + 轻量部署**

我们的护城河：
1. **44 平台全覆盖**的热榜采集（BettaFish 只有 10+）
2. **审查检测**是独有功能（BettaFish 不做删帖检测）
3. **轻量单体**，SQLite + FastAPI，一台机器就能跑
4. **5 分钟闪删检测**，接近实时

不要盲目追 BettaFish 的全功能，而是在自己的垂直方向做深。
